# Plan 62-01: Guarded Position Exit Execution

<objective>
Harden `ops position-adjustment-execute` so operator-confirmed close/reduce
execution is token-gated, service-safe, filter-aware, manual-symbol-safe, and
post-action verified before protective cleanup.
</objective>

<tasks>

1. Preserve and test confirmation gates.
   - Ensure live mode, live-service inactive state, fresh token matching,
     no-confirmed-`now`, and symbol-filter requirements continue to block
     execution before any order call.
   - Add a manual-only position test proving no manual adjustment order can be
     executed.

2. Verify partial-reduce post state.
   - After partial reduce submission, read current position amount.
   - If post amount is missing or not reduced to
     `expected_remaining_position_amt` or lower in absolute value, mark the
     execution as `position_adjustment_submitted_cleanup_deferred`.
   - Persist the post-check error and do not claim a clean completion.

3. Guard full-close protective cleanup.
   - Keep requiring the relevant position side to be flat before cleanup.
   - Read open algo orders for the symbol before calling the symbol-wide
     cancel endpoint.
   - Defer cleanup if any open algo order belongs to a different position side.

4. Verify locally and on server.
   - Run focused position-adjustment and CLI tests.
   - Run full local suite.
   - Deploy to `/opt/binance-futures-agent` only after local tests pass.
   - Run server focused/full tests and a preview-only
     `position-adjustment-execute` smoke without a confirmation token.

5. Close GSD phase.
   - Write summary and verification.
   - Update ROADMAP, REQUIREMENTS, STATE, and PROJECT.
   - Commit the Phase 62 implementation and closeout.

</tasks>

<success_criteria>

- Confirmed execution refuses to place orders without a fresh matching token.
- Manual positions never produce executable adjustment orders.
- Confirmed execution refuses when service-active, filters missing, or live mode
  is not active.
- Partial reduce post-action verification catches insufficient reductions.
- Full-close cleanup is deferred when cross-side algo orders make symbol-wide
  cancel unsafe.
- No verification step places or cancels live Binance orders.
</success_criteria>
