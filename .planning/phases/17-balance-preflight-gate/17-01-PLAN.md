# Plan 17-01: Balance Preflight Gate

## Objective

Avoid repeated live entry attempts when the Binance USD-M futures account has
less available balance than the order intent's estimated initial margin.

## Triggering Evidence

After Phase 16 deployed cross margin and hedge position mode, a live HYPEUSDT
trade decision reached entry order placement. Binance rejected the entry with
`-2019 Margin is insufficient`. A read-only futures account check showed
`availableBalance=0.00000000`, so the bot should reject locally before margin
setup or entry order calls until the USD-M futures account is funded.

## Scope

- In live mode, read Binance account `availableBalance` after deterministic risk
  gates and before margin setup.
- Compare `availableBalance` with `OrderIntent.estimated_initial_margin_usdt`.
- If the balance is unknown, insufficient, or the account read fails, return a
  rejected, non-submitted execution result.
- Preserve dry-run and testnet behavior.
- Preserve all pilot caps:
  - 100 USDT account capital;
  - 20 USDT max position notional;
  - 3x max leverage;
  - 1 USDT max risk per trade;
  - 3 USDT max daily loss.

## Verification

- Execution regression covers insufficient available balance rejecting before
  margin, leverage, or order calls.
- Execution regression covers account balance API errors rejecting before
  margin, leverage, or order calls.
- Existing live accepted-path tests confirm funded accounts still proceed to
  margin setup, leverage setup, entry, and protective orders.
- Full unit suite passes.
- Server deployment observes the unfunded live account rejecting safely without
  submitting an entry order.

## Safety Notes

This gate does not make the strategy more aggressive. It reduces exchange-side
error churn and makes the current unfunded-account state explicit. Real entry
submission still requires USD-M futures available balance plus all prior risk,
margin, position-mode, and protective-order gates.
