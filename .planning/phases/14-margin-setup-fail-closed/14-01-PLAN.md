# Plan 14-01: Margin Setup Fail-Closed

## Objective

Handle Binance margin/leverage setup failures as fail-closed rejected execution
results instead of uncaught live service crashes.

## Triggering Evidence

After Phase 13 expanded the pilot universe, the live timer selected ZECUSDT and
the AI returned a valid `trade` decision. Before any entry order was submitted,
Binance rejected `POST /fapi/v1/marginType` because the account is in
Multi-Assets mode and cannot switch the symbol to isolated margin. The exception
was uncaught, causing the systemd one-cycle service to fail.

## Scope

- Add a regression test reproducing Multi-Assets mode isolated-margin rejection.
- Catch `BinanceSignedError` from pre-entry margin/leverage setup.
- Return `status=rejected`, `submitted=false`, and `margin_setup_failed`.
- Persist the rejected order intent and margin error evidence.
- Do not submit entry orders when margin setup fails.
- Keep all live risk caps and protective-order requirements unchanged.

## Verification

- Regression test fails before the fix and passes after the fix.
- Execution-engine tests pass.
- Full unit suite passes.
- Server deploy health check passes.
- A post-deploy live cycle no longer crashes on the same margin setup condition.

## Safety Notes

This phase chooses safety over forcing trades under account mode incompatibility.
If the operator wants actual isolated-margin live entries, the Binance account
mode must be changed outside the bot, or a separate reviewed phase must approve
cross/multi-assets execution semantics.
