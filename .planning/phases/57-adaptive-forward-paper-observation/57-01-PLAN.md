# Plan 57-01: Adaptive Forward-Paper Observation

<objective>
Make guarded forward-paper runs diagnosable even when strict setup or guard
rules produce few or zero signals. The phase must keep live trading disabled and
must not create order intents, but every broad hot-symbol observation should
explain what happened.
</objective>

<tasks>

1. Add paper observation persistence.
   - Add a `paper_observations` event-store category.
   - Record one observation per observed symbol per forward-paper run.
   - Persist generated signals, setup passes, guard blocks, open-signal skips,
     insufficient bars, and incomplete setups with reason codes.

2. Preserve setup factor snapshots for rejected candidates.
   - Store setup summary, factor scores, full setup payload, bars seen, status,
     and observation timestamp.
   - Reuse existing deterministic `build_trade_setup` logic so paper
     observation matches live/backtest setup geometry.

3. Add source-health reporting.
   - Report whether symbols came from explicit CLI symbols, Binance 24h ticker
     auto-hot selection, or configured fallback symbols.
   - Include auto-hot filters, selected rows, selected symbols, and payload
     counts.
   - Check event-store narrative rows for selected-symbol coverage and source
     counts so Square/manual/RSS exports are visible when present.

4. Keep the live boundary closed.
   - Do not write `order_intents`.
   - Do not call signed Binance endpoints.
   - Do not change live env, timers, services, risk profile, positions, or
     orders.

5. Add tests and smoke evidence.
   - Unit-test generated signal observations.
   - Unit-test guard block observations.
   - Unit-test setup-pass observations with factor snapshots.
   - Unit-test event-store narrative source-health reporting.
   - CLI-test source-health payloads.
   - Run a public-market paper-only smoke over 40 auto-hot symbols and verify
     `order_intents=0`.

</tasks>

<success_criteria>

- Forward-paper runs persist observation evidence for generated and rejected
  candidates.
- Zero-signal or low-signal runs expose skip reasons and guard/setup factor
  context.
- Auto-hot observation can cover 40 Binance USD-M symbols while remaining
  paper-only.
- Source-health output explains symbol-selection source and narrative event
  coverage.
- No live order intent, server runtime mutation, or exchange mutation is
  introduced.
</success_criteria>
