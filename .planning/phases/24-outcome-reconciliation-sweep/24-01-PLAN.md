# Plan 24-01: Outcome Reconciliation Sweep

## Objective

Add an operator command that sweeps submitted live order intents, reconstructs
their Binance `userTrades` outcomes, and persists only final closed outcomes so
risk-profile changes can be unblocked without one-off manual symbol commands.

## Scope

- Add an `ops reconcile-outcomes` command.
- Load submitted order intents from the existing SQLite event store.
- Skip submitted intents that already have `outcome:{event_id}:closed` unless
  explicitly requested.
- Fetch read-only Binance `userTrades` for each checked intent.
- Summarize each intent as `closed` or `open_or_partial` using existing fill
  accounting.
- With `--persist-closed`, write fills and outcomes only for `closed` results.
  Do not persist `open_or_partial` outcomes by default.
- Add unit and CLI tests covering closed-only persistence and default skipping
  of already reconciled intents.

## Verification

- Focused outcome/CLI tests cover sweep behavior.
- Full local test suite passes.
- Server focused tests pass after deployment.
- Server `ops reconcile-outcomes --persist-closed` runs against live data
  without changing exchange state; while BNBUSDT remains open, it should report
  `open_or_partial` and insert no BNB outcome.
- Server `ops risk-change-check --target-leverage 8` remains blocked until all
  active positions clear and final closed outcomes are persisted.

## Safety Notes

This phase must not close positions, cancel orders, change leverage, edit server
env risk caps, or disable/enable systemd timers. It may persist local fill and
outcome artifacts only when Binance fills summarize to a final closed trade.
