# Plan 55-01: Operator Resume Decision Packet

<objective>
Add a single read-only operator decision packet that synthesizes existing
live-resume readiness evidence into a clear next action without mutating server,
exchange, env, timer, profile, or order state.
</objective>

<tasks>

1. Add decision synthesis module.
   - Create `src/bfa/ops/operator_resume_decision.py`.
   - Define a dataclass with schema `bfa_operator_resume_decision_v1`.
   - Accept an existing readiness report object or mapping and classify the
     decision status as one of `keep_live_paused`, `collect_more_paper`,
     `resolve_exposure`, or `eligible_for_operator_resume`.
   - Group blockers into strategy, paper, server, exchange/manual exposure,
     risk profile, AI/provider health, and confirmation.
   - Include read-only mutation flags set to false.

2. Add CLI command.
   - Add `ops operator-resume-decision`.
   - Support `--readiness-report` for a JSON readiness artifact.
   - When no readiness artifact is supplied, reuse the same arguments as
     `ops live-resume-readiness` to build a fresh read-only readiness report.
   - Return exit code `0` only for `eligible_for_operator_resume`; otherwise
     return `1`.

3. Add tests.
   - Unit-test status priority for manual/unattributed exposure.
   - Unit-test paper-only blockers returning `collect_more_paper`.
   - Unit-test confirmation-only readiness returning
     `eligible_for_operator_resume`.
   - CLI smoke-test `--readiness-report` output and non-zero exit while blocked.

4. Update docs and GSD closeout.
   - Document the server command in `docs/deployment.md`.
   - Write Phase 55 summary and verification artifacts.
   - Update `ROADMAP.md`, `STATE.md`, `PROJECT.md`, and `REQUIREMENTS.md`
     after verification.

</tasks>

<verification>

- `python -m unittest tests.test_ops_operator_resume_decision tests.test_cli`
- `python -m unittest discover -s tests`
- `git diff --check`
- Secret-pattern scan on changed tracked files.

</verification>

<success_criteria>

- Decision packet emits exactly one of the four required statuses.
- Blockers are grouped across strategy, paper, server, exchange/manual
  exposure, risk profile, AI/provider health, and confirmation.
- Packet and CLI remain read-only and expose no mutation-capable behavior.
- Eligibility still routes to a separate operator confirmation step and never
  resumes live automation.
</success_criteria>
