# Plan 53-01: Server Live-Resume Readiness Evidence

<objective>
Add and use a safe server-readiness evidence path that runs the Phase 52
`ops live-resume-readiness` command read-only, treats manual ETH/ETHUSDT as
manual exposure, captures a secret-safe artifact, and does not restore live
automation or mutate Binance/server risk state.
</objective>

<tasks>

1. Add a server readiness helper script.
   - Create `scripts/run-server-readiness.ps1`.
   - Default to preview mode; require an explicit execution flag before
     connecting to the server.
   - Restrict remote paths to `/opt/binance-futures-agent` and
     `/etc/binance-futures-agent`.
   - Build a remote command that runs:
     `python -m bfa.cli ops live-resume-readiness`
     with `--env-file`, `--db`, `--variant quant_setup_selective_guarded`, and
     `--manual-exposure-symbols ETHUSDT` by default.
   - Accept exit code `0` or `1` only when stdout parses as
     `bfa_live_resume_readiness_v1`.
   - Write the JSON stdout to a local `runtime/` evidence file when execution is
     requested.

2. Prove the helper is operationally safe.
   - Extend `tests/test_deploy_assets.py` to verify the helper defaults to
     preview mode.
   - Verify it references only isolated server paths.
   - Verify it includes the manual exposure flag.
   - Verify it does not contain commands that enable/start live timers,
     services, profile apply commands, or order execution commands.
   - Verify no secret literals are embedded.

3. Document the read-only procedure.
   - Update `docs/deployment.md` with a "Live Resume Readiness" section.
   - Show preview and execution commands.
   - Explain that exit code `1` can be an expected fail-closed readiness result.
   - State explicitly that the procedure does not restore timers, apply risk
     profiles, write env files, or place/cancel orders.

4. Run local verification.
   - Run focused deploy asset tests.
   - Run existing live-resume readiness tests.
   - Run `git diff --check`.
   - Run a changed-file secret scan.

5. Attempt server read-only evidence collection when SSH auth is available.
   - Use batch-mode SSH first to avoid hanging on password prompts.
   - If SSH is unavailable, record the blocker in the Phase 53 summary and keep
     the phase open or mark server evidence pending.
   - If SSH is available, run the helper with explicit execution enabled and
     save the JSON artifact under local `runtime/`.
   - Inspect only secret-safe fields: schema, status, read_only flags,
     server_state, exchange/manual exposure classification, and reasons.
   - Do not run any command that starts/enables live timer/service, applies a
     risk profile, edits `/etc/binance-futures-agent/env`, or submits/cancels
     Binance orders.

</tasks>

<verification>

- `python -m unittest tests.test_deploy_assets`
- `python -m unittest tests.test_ops_live_resume_readiness`
- `git diff --check`
- Changed-file secret scan returns no hits.
- If server SSH is available, the readiness artifact has:
  - `schema == "bfa_live_resume_readiness_v1"`
  - `read_only.places_orders == false`
  - `read_only.applies_risk_profiles == false`
  - `read_only.writes_env_files == false`
  - `read_only.changes_systemd_state == false`
  - `read_only.mutates_exchange_state == false`
  - manual ETH/ETHUSDT exposure classified separately when present.

</verification>

<success_criteria>

- Server readiness can be previewed and run through a dedicated helper without
  live-mutating commands.
- Secret-safe readiness evidence is captured when server access is available.
- Manual ETH/ETHUSDT exposure is represented as manual exposure, not
  agent-managed evidence.
- Verification proves the helper and command do not place, cancel, or modify
  Binance orders; do not apply risk profiles; and do not restore live timers.

</success_criteria>
