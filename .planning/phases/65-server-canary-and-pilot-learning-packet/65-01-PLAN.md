# Plan 65-01: Server Canary And Pilot Learning Packet

<objective>
Add and deploy a read-only pilot learning packet command that composes current
server lifecycle, cap usage, exit, live outcome, guard-feedback, and trace
evidence into one artifact, then verify the isolated server canary with timers
restored.
</objective>

<tasks>

1. Build the packet module.
   - Read first: `src/bfa/ops/exposure_status.py`,
     `src/bfa/ops/position_review.py`,
     `src/bfa/ops/position_hold_check.py`,
     `src/bfa/ops/live_outcome_ledger.py`, and `src/bfa/ops/trade_trace.py`.
   - Add `src/bfa/ops/pilot_learning_packet.py`.
   - Implement `build_pilot_learning_packet_report(config, db_path=None,
     check_binance=True, signed_client=None, now=None, latest_traces=5)`.
   - Compose existing source report builders; do not duplicate Binance order or
     risk logic.
   - Emit `schema=bfa_pilot_learning_packet_v1`.

2. Summarize operator evidence.
   - Include current manual symbols from `BFA_MANUAL_POSITION_SYMBOLS`.
   - Include lifecycle decisions from position review, preserving
     `manual_hold` and `manual_position_ignored`.
   - Include cap usage from exposure status: max/current profile, current
     sizing, entry capacity, active bot exposure, manual exposure, available
     balance, total wallet balance, and cap-utilization ratios.
   - Include exit-plan status/reasons and per-position plan blockers.
   - Include live outcome ledger summary, latest outcomes, and
     recommendation-only guard feedback.
   - Include bounded trade traces for active matching intent IDs and latest
     outcome intent IDs.

3. Enforce non-mutation proof.
   - Packet `mutation_proof` must show:
     `places_orders=false`, `cancels_orders=false`,
     `changes_systemd_state=false`, `writes_env_files=false`,
     `raises_risk=false`, `applies_guard_changes=false`,
     `persists_closed_fills_and_outcomes=false`.
   - The packet command must not expose confirmation tokens or secret values.

4. Add CLI wiring and docs.
   - Add `ops pilot-learning-packet` to `src/bfa/cli.py`.
   - Flags: `--env-file`, `--db`, `--now`, `--latest-traces`,
     `--skip-binance`.
   - Return JSON to stdout and exit `0` when the packet is produced.
   - Add README usage for generating `runtime/phase65-pilot-learning-packet.json`.

5. Add tests.
   - Add `tests/test_ops_pilot_learning_packet.py` with fake signed-client
     coverage for active manual exposure, cap usage, lifecycle decisions,
     exit status, live ledger, trade traces, and mutation proof.
   - Extend `tests/test_cli.py` to assert CLI JSON shape for the new command.

6. Run local verification.
   - Run focused tests:
     `python -m unittest tests.test_ops_pilot_learning_packet tests.test_cli`.
   - Run full tests: `python -m unittest discover -s tests`.
   - Run `git diff --check`.

7. Deploy and generate server canary artifacts.
   - Pause only `binance-futures-agent-live.timer` and
     `binance-futures-agent-paper.timer` if deployment needs file sync.
   - Deploy only to `/opt/binance-futures-agent/app` and
     `/etc/binance-futures-agent`.
   - Run server focused and full tests.
   - Run server `ops pilot-learning-packet` with signed read-only Binance
     evidence and write
     `/opt/binance-futures-agent/app/runtime/phase65-pilot-learning-packet.json`.
   - Run server health-check and timer state checks.
   - Restore live and paper timers; verify timers active and services inactive.

8. Close GSD phase and milestone.
   - Write `65-01-SUMMARY.md` and `65-VERIFICATION.md` with local/server
     command evidence.
   - Update `.planning/ROADMAP.md`, `.planning/REQUIREMENTS.md`,
     `.planning/STATE.md`, and `.planning/PROJECT.md`.
   - Commit implementation and closeout.

</tasks>
