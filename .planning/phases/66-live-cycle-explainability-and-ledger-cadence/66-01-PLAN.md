# Plan 66-01: Live Cycle Explainability And Ledger Cadence

<objective>
Add a read-only live-cycle explainability command that summarizes recent
trade/no-trade cycles, explains sizing/risk constraints, and can run the live
ledger cadence path without mutating Binance orders, env files, systemd state,
or guard/risk configuration.
</objective>

<tasks>

1. Build the live-cycle explainability module.
   - Read first: `src/bfa/agent.py`, `src/bfa/ops/trade_trace.py`,
     `src/bfa/ops/live_outcome_ledger.py`,
     `src/bfa/ops/pilot_learning_packet.py`,
     `src/bfa/execution/sizing.py`, and `src/bfa/execution/risk.py`.
   - Add `src/bfa/ops/live_cycle_explainability.py`.
   - Implement `build_live_cycle_explainability_report(config, db_path=None,
     since=None, latest_cycles=10, include_ledger=True, reconcile=False,
     persist_closed=False, signed_client=None)`.
   - Emit `schema=bfa_live_cycle_explainability_v1`.

2. Reconstruct recent cycle evidence.
   - Query recent `position_lifecycle_decision` artifacts from `risk_states`.
   - Query recent `candidates`, `trade_setups`, `ai_decisions`,
     `order_intents`, `exchange_responses`, and `outcomes`.
   - Group artifacts by symbol plus decision timestamp/ref ID where possible.
   - Include `evidence_quality` notes when no stable grouping is available.
   - Include submitted and no-order cycles; do not limit the report to
     submitted order intents.

3. Explain trade/no-trade and sizing decisions.
   - For each cycle, include evaluated symbol, side, candidate score, setup
     decision, factor scores/reasons, AI validation result, risk acceptance,
     risk rejection reasons, submitted status, and exchange response status.
   - Add `sizing_explanation` using available trade setup, AI decision, risk
     state, dynamic sizing result, and config caps.
   - Name limiting factors such as `stop_risk_cap`, `margin_fraction_cap`,
     `effective_notional_cap`, `below_min_executable_notional`,
     `risk_exceeds_cap`, and portfolio cap reasons when present.
   - Preserve manual symbols in diagnostics while marking them non-bot-managed.

4. Add ledger cadence integration.
   - Reuse `build_live_outcome_ledger_report()`.
   - Default to DB-only ledger reads.
   - Allow `--reconcile` and `--persist-closed` with the same guard behavior as
     `ops live-outcome-ledger`.
   - Include a clear `mutation_proof` block distinguishing:
     exchange mutation, env/systemd/risk/guard mutation, and optional local
     event-store outcome persistence.

5. Add CLI wiring and documentation.
   - Add `ops live-cycle-explainability` to `src/bfa/cli.py`.
   - Flags: `--env-file`, `--db`, `--since`, `--latest-cycles`,
     `--no-ledger`, `--reconcile`, `--persist-closed`.
   - Use signed Binance client only when reconciliation is requested.
   - Add README usage and explain how to read small-position sizing reasons.

6. Add tests.
   - Add `tests/test_ops_live_cycle_explainability.py`.
   - Cover submitted cycle trace, no-order/AI-pass cycle, quant/risk-blocked
     cycle, missing-artifact evidence-quality notes, manual-symbol visibility,
     sizing explanation, ledger inclusion, and mutation proof.
   - Extend `tests/test_cli.py` for CLI JSON shape and reconcile flag plumbing.

7. Run local verification.
   - Run focused tests:
     `python -m unittest tests.test_ops_live_cycle_explainability tests.test_cli`.
   - Run full tests: `python -m unittest discover -s tests`.
   - Run `git diff --check`.
   - Run `node $HOME/.codex/gsd-core/bin/gsd-tools.cjs query audit-open --json`.

8. Close GSD phase.
   - Write `66-01-SUMMARY.md` and `66-VERIFICATION.md` with command evidence.
   - Update `.planning/ROADMAP.md`, `.planning/REQUIREMENTS.md`,
     `.planning/STATE.md`, and `.planning/PROJECT.md`.
   - Commit implementation and closeout.

</tasks>

  skip reasons, factor scores, AI decisions, risk decisions, sizing caps, and
  submitted-order status.

  margin cap, filters, or portfolio caps constrain size.
- Closed live outcome reconciliation and ledger reporting can run on a single
  scheduled/single-command path with mutation proof showing no order/env/systemd
  or risk/guard application.
- Manual symbols remain visible in diagnostics without being treated as
  bot-managed evidence.
- Phase 66 covers OPS-03 and LEARN-04.
